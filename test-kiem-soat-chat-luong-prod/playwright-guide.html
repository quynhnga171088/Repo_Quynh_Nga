1.6 Thực thi test tự động trên Kari Inspector

Sau khi đã verify toàn bộ test case trên local thành công và script đã được merge vào branch main, thực hiện các bước sau để chạy regression test trực tiếp trên Kari Inspector.

Điều kiện trước khi thực thi

Điều kiệnMô tả
Code đã implement đầy đủTất cả test case trong Execution đã được viết code và chạy pass trên local
Script đã merge vào mainPull Request đã được review và merge — không còn ở nhánh feature
Kari Inspector đã kết nối BitbucketCấu hình ở mục 1.2 đã hoàn tất, trạng thái "Bitbucket đã kết nối"
⚠️
Tại sao phải merge vào main trước?

Kari Inspector pull code từ branch đã cấu hình (mặc định là main) để chạy test — script còn trên nhánh feature sẽ không được nhận diện.

1
Truy cập Thực thi IT và chọn Execution

Trên thanh menu trái của Kari Inspector, click vào Thực thi IT. Trong danh sách, click vào tên Execution đã tạo trước đó (ví dụ: Manual Test — Sprint 13).

Màn hình mở ra hiển thị danh sách các test case automation sẽ được thực thi.

2
Nhấn nút Run Testcase

Click nút Run Testcase (hoặc Chạy automation cho app này) để bắt đầu chạy toàn bộ test case trong Execution.

3
Hệ thống tự động thực thi

Sau khi nhấn Run, hệ thống tự động:

1. Pull code mới nhất từ Bitbucket
→
2. Chạy toàn bộ test case
3. Ghi nhận Pass / Fail từng test
→
4. Hiển thị báo cáo tổng hợp ✅

Khi nào nên dùng tính năng Thực thi test?

🎯
Mục đích: Test Regression định kỳ

Tính năng này đảm bảo logic cũ của hệ thống không bị ảnh hưởng trong quá trình phát triển tính năng mới.

Tình huốngMô tả
Sau mỗi sprint / releaseChạy toàn bộ regression để xác nhận không có regression bug
Trước khi deploy staging / productionKiểm tra lần cuối trước khi phát hành
Khi team dev merge code lớn vào mainPhát hiện sớm các thay đổi vô tình làm gãy chức năng cũ
Theo lịch định kỳ (hàng tuần / hàng ngày)Monitoring liên tục chất lượng hệ thống
💡
Phân biệt: Tạo Execution vs Thực thi test

Tạo Execution (mục 1.4) thực hiện một lần để gen code từ testcase manual. Thực thi test (mục này) là việc chạy lại nhiều lần trên code đã hoàn thiện để regression test.

Phần 3

🤖 Dùng Cursor AI để sửa test sai / bị lỗi

Hướng dẫn cách dùng Cursor AI để phát hiện và sửa các test bị lỗi — chỉ cần mô tả vấn đề bằng tiếng Việt, AI sẽ gợi ý và thực hiện sửa code.

Khi nào cần sửa test?

Thông báo lỗiÝ nghĩa
Locator not foundSelector không đúng với giao diện thực tế
Timeout exceededTrang load chậm hoặc click/fill sai element
Expected ... to be visibleAssertion kiểm tra sai text/element
Navigation failedURL trong test không đúng
test.skip(...)Test chưa có code thực hiện — chỉ có comment
💡
Lưu ý

Test sinh tự động thường đúng về nghiệp vụ (mô tả Steps, Expected Result) nhưng có thể sai ở phần code tương tác với UI thực tế.

Cấu trúc một test case

TypeScript — .spec.ts
test('DEMO_01_054: Đăng nhập - Thành công với thông tin hợp lệ', async ({ page }) => { // **Preconditions:** Tài khoản đã tồn tại. Ở trang My Account. // **Steps:** // 1. Nhập username hợp lệ vào trường Username // 2. Nhập password hợp lệ vào trường Password // 3. Nhấn nút Login // **Expected Result:** Đăng nhập thành công, chuyển hướng đến Dashboard. await page.goto('https://practice.automationtesting.in/my-account/'); await page.locator('#username').fill('user@example.com'); await page.locator('#password').fill('password123'); await page.getByRole('button', { name: 'Login' }).click(); await expect(page.getByRole('link', { name: 'Logout' })).toBeVisible(); });

Quy trình sửa test — 6 bước

1
Xác định test lỗi

Chạy npm test → xem báo cáo npx playwright show-report → ghi chú tên test bị failed.

2
Mở AI Chat

Nhấn Ctrl+L (Windows) hoặc Cmd+L (macOS).

3
Thêm context file test

Gõ @ rồi gõ tên file (ví dụ @xác_thực_và_tài_khoản.spec.ts) → chọn từ dropdown.

4
Mô tả lỗi cho AI

Dùng prompt mẫu phía dưới theo loại lỗi đang gặp.

5
Áp dụng gợi ý

Đọc lướt code AI đề xuất → nhấn Apply.

6
Chạy lại để kiểm tra
Terminal
npx playwright test tests/xác_thực_và_tài_khoản/ --headed

✅ passed → Thành công!   ❌ failed → Quay lại Bước 4 với lỗi mới.

Prompt mẫu theo từng loại lỗi

Lỗi Locator not found / Timeout

💬 Prompt mẫu — copy & paste vào Cursor AI
Test DEMO_01_054 bị lỗi "locator not found" khi chạy trên trang https://practice.automationtesting.in/my-account/ Thông báo lỗi: [Paste thông báo lỗi từ báo cáo vào đây] Hãy kiểm tra và sửa lại selector/locator cho đúng với giao diện thực tế. Giữ nguyên logic test và comment mô tả các bước.

Lỗi Assertion sai (Expected to be visible)

💬 Prompt mẫu
Test DEMO_01_054 bị lỗi ở phần kiểm tra kết quả (assertion). Thông báo lỗi: [Paste lỗi vào đây] Trang sau khi đăng nhập hiển thị text "Hello [tên user]" thay vì "Dashboard". Hãy sửa lại assertion cho khớp với text thực tế hiển thị trên trang.

3
Thao tác trên trình duyệt

Thực hiện đúng các bước của test case (click, nhập liệu…). Mỗi thao tác sẽ tự thêm một dòng code vào Inspector.

4

🎯 Chiến lược xác định Locator trong Playwright

Locator là bộ chỉ dẫn để Playwright tìm và tương tác với một phần tử trên trang web. Chọn locator đúng là cốt lõi để test không bị gãy khi giao diện thay đổi.

6.1 Kim tự tháp ưu tiên Locator (9 cấp)

Chọn locator từ trên xuống dưới. Chỉ xuống cấp khi cấp trên không áp dụng được.

① getByRole() ← Ưu tiên cao nhất — Tiêu chuẩn vàng ② getByLabel() ← Dành cho phần tử form ③ getByPlaceholder() ← Khi không có label ④ getByText() ← Theo văn bản hiển thị ⑤ getByAltText() ← Tìm hình ảnh ⑥ getByTitle() ← Tìm qua tooltip ⑦ getByTestId() ← Cần dev bổ sung data-testid ⑧ CSS Selector ← Phương án dự phòng ⑨ XPath ← Phương án cuối cùng

6.2 Nhóm 1 — Locator hướng tới người dùng

① getByRole() — Tiêu chuẩn vàng

Tìm phần tử theo vai trò ngữ nghĩa — giống cách người dùng và trình đọc màn hình nhận diện. Bền vững nhất, không phụ thuộc class/id.

TypeScript
page.getByRole('button', { name: 'Đăng nhập' }) page.getByRole('textbox', { name: 'Email' }) page.getByRole('heading', { level: 1 }) page.getByRole('link', { name: 'Xem sản phẩm' }) page.getByRole('checkbox') page.getByRole('combobox') // select dropdown
📋 Bảng tra cứu vai trò phổ biến▼
Thẻ HTMLVai trò (Role)
<button>, <input type="submit">button
<a href="...">link
<input type="text">, <textarea>textbox
<input type="checkbox">checkbox
<input type="radio">radio
<select>combobox
<h1> – <h6>heading
<nav>navigation
<img alt="...">img

② getByLabel() — Dành cho phần tử form

TypeScript
await page.getByLabel('Địa chỉ email:').fill('test@example.com'); await page.getByLabel('Mật khẩu').fill('123456');

③ getByPlaceholder()  |  ④ getByText()  |  ⑤ getByAltText()

TypeScript
// ③ getByPlaceholder — khi không có label await page.getByPlaceholder('Tìm kiếm sản phẩm...').fill('iPhone 15'); // ④ getByText — kiểm tra thông báo, nhãn tĩnh await expect(page.getByText('Tên đăng nhập hoặc mật khẩu không đúng.')).toBeVisible(); // ⑤ getByAltText — tìm hình ảnh await expect(page.getByAltText('Logo cửa hàng')).toBeVisible();
⚠️
getByText() dễ bị lỗi khi nội dung thay đổi

Dùng { exact: true } để tăng độ chính xác. Tránh dùng cho các đoạn text dài hoặc có thể bị dịch sang ngôn ngữ khác.

6.3 Nhóm 2 — Locator kỹ thuật (Phương án dự phòng)

⑧ CSS Selector

TypeScript — CSS Selector
page.locator('#login-button') // Theo ID — bền vững nhất page.locator('.error-message') // Theo class page.locator("input[name='email']") // Theo thuộc tính page.locator("button.btn-primary[type='submit']") // Kết hợp page.locator("[id^='user-']") // ID bắt đầu bằng 'user-' page.locator("[id*='-profile-']") // ID chứa '-profile-'
Quy tắc viết CSS bền vữngVí dụ
✅ Ưu tiên ID và data-testid#user-btn, [data-testid="save-btn"]
❌ Tránh selector dài nhiều cấpdiv > div > section > button
✅ Dùng "mỏ neo" (anchor)[data-username='bob'] button.delete-btn
❌ Tránh class tự động sinh.button-a1B2c_root

⑨ XPath — Vũ khí cuối cùng

XPath có thể di chuyển lên (cha), xuống (con), và ngang (anh em) trong cây DOM — điều CSS không làm được. Chỉ dùng khi tất cả cách khác đã thất bại.

TypeScript — XPath
page.locator("//button[text()='Đăng nhập']") page.locator("//div[contains(@class, 'error-message')]") page.locator("//label[text()='Email']/following-sibling::input") page.locator("//span[text()='Alice']/parent::div")
📋 Bảng tra cứu trục XPath▼
Trục XPathÝ nghĩaVí dụ
parent:: hoặc ..Lên phần tử cha//span[text()='Alice']/parent::div
following-sibling::Anh em đứng sau//label[text()='Email']/following-sibling::input
preceding-sibling::Anh em đứng trước//label[text()='Điều khoản']/preceding-sibling::input
ancestor::Tổ tiên bất kỳ cấp//button[text()='Gửi']/ancestor::form
🚨
KHÔNG dùng Absolute XPath

/html/body/div[1]/div[2]/button[3] — chỉ cần dev thêm một thẻ div, toàn bộ đường dẫn sẽ sai.

6.4 Nối chuỗi (Chaining) & Lọc (.filter())

Chaining — Nối chuỗi locator

TypeScript
// ❌ Một chuỗi dài — khó đọc page.locator("div[data-username='bob'] button.delete-btn"); // ✅ Nối chuỗi — dễ hiểu và tái sử dụng const bobRow = page.locator("div[data-username='bob']"); const deleteBtn = bobRow.getByRole('button', { name: 'Xóa' }); await deleteBtn.click(); // Tái sử dụng bobRow const bobName = bobRow.locator('span');

.filter() — Lọc trong danh sách

TypeScript
// Bước 1: Chọn toàn bộ nhóm const listItems = page.getByRole('listitem'); // Bước 2: Lọc ra mục chứa text 'Bob' const bobItem = listItems.filter({ hasText: 'Bob' }); // Bước 3: Tương tác với phần tử bên trong await bobItem.getByRole('button', { name: 'Chỉnh sửa' }).click();
